# 跨系统表格数据转换智能体 — MorphSheet 项目报告

---

## 一、项目背景

### 1.1 问题背景

企业内部不同业务系统（ERP、CRM、老旧自研系统）的数据结构与标准往往不互通。当需要将 A 系统的数据导入 B 系统时，IT 人员通常面临以下痛点：

- **格式不兼容**：源文件可能是 Excel 2007 (.xlsx)，目标系统只接受 Excel 97-2003 (.xls) 或 GBK 编码的 CSV
- **字段映射复杂**："姓名" 要映射为 "Emp_Name"，日期格式从 `YYYY/MM/DD` 改为 `YYYYMMDD`
- **脏数据频发**：电话号码含括号、连字符，金额带 ¥ 符号，日期值超出合理范围（如 99999999）
- **重复劳动**：每次换一个部门或供应商的导出文件，又要手动重做一遍清洗流程

传统的解决方案是编写 Excel 宏（Macro）或固定 Python 脚本，但这些方案**缺乏对未知表格格式的语义理解能力**——宏和脚本只能处理预定义的列名和格式。

### 1.2 项目定位

MorphSheet 是一款**基于大模型 Agent 的桌面端数据中枢**。它超越静态脚本，具备对未知表格格式的语义理解能力。用户通过自然语言下达清洗指令，Agent 自动分析源数据的 Schema（列名、数据类型、格式分布），调用 LLM 动态生成 Pandas 转换代码，在安全沙箱中执行，并将成功的转换逻辑沉淀为可复用的"技能模板"。

### 1.3 项目概述

| 维度 | 说明 |
|------|------|
| 项目名 | MorphSheet — 跨系统表格数据转换智能体 |
| 平台 | Windows 桌面应用 (PyWebView + FastAPI + HTML/CSS/JS) |
| 核心 LLM | DeepSeek (deepseek-chat) |
| 数据处理 | Pandas / openpyxl / xlrd / xlwt / chardet |
| 规模 | ~2600 行 Python + ~2000 行前端，30 个 Python 模块 |
| 开发周期 | 7 阶段迭代，38 次提交 |

### 2.3 关键技术细节

#### 2.3.1 Schema 与格式模式提取

系统不仅提取列名和数据类型，还会对每列前 50 行进行**格式模式分析**，识别出日期、电话、货币等 12 种格式类型的分布比例。例如入职日期列可能被识别为：

| 格式 | 占比 | 示例 |
|------|------|------|
| YYYY-MM-DD | 44% | 2023-02-01 |
| YYYY/MM/DD | 30% | 2018/10/14 |
| YYYY年M月D日 | 14% | 2022年8月3日 |
| YYYYMMDD | 12% | 20230523 |

这些信息作为 `column_patterns` 注入 Prompt，确保 LLM 了解数据中的所有格式变体。

#### 2.3.2 安全沙箱

动态生成的代码具有不可控性。系统采用三层防护：

1. **AST 静态扫描**：解析代码 AST，拦截 `os`/`subprocess`/`sys`/`eval`/`exec`/`open` 等 50+ 危险模块和函数调用
2. **受限命名空间**：`exec()` 执行时仅注入安全的内置函数（无 `__import__` 逃逸路径）和 `pandas`/`numpy`/`re`/`datetime` 四个白名单库
3. **超时控制**：代码在独立线程中运行，30 秒超时自动终止

#### 2.3.3 脏数据检测（三层扫描）

| 层级 | 检测内容 | 机制 |
|------|---------|------|
| **预扫描** | 源数据中的明显非法值（99999999 日期等） | LLM 执行前独立扫描，使用原始行号和真实值 |
| **NaN 检测** | 转换后变为空的值 | `pd.isna()` + `'NaT'` 字符串匹配 |
| **格式不匹配** | 值存在但格式不符合主导模式 | 分析列的主导格式（>60%），标记异常值 |

预扫描使用 `actionable=False` 标记（仅展示信息），避免原始行号与结果行号错位导致误删正常数据。后置扫描使用 `actionable=True` 标记（可操作），通过 CUST_ID 等标识列精确定位到结果行进行删除。

#### 2.3.4 格式降级与编码兼容

- **xls 降级**：xlwt 直接写入（Pandas 2.0+ 不再支持 xlwt 引擎），自动处理 65535 行截断和 256 列限制
- **CSV BOM**：UTF-8 文件自动添加 `\xef\xbb\xbf` BOM 头，确保 Windows 记事本和老旧系统可识别
- **GBK 编码**：chardet 自动检测源文件编码，支持 GBK/GB2312/UTF-8/latin-1 多编码回退链

#### 2.3.5 技能记忆系统

```
用户完成转换 → 点击"保存为技能"
     │
     ▼
SQLite 存储: {源Schema + 目标格式 + Pandas代码 + 列映射}
     +
```

## 四、作品的主要亮点

### 4.1 技术创新点

**（1）自然语言驱动的动态代码生成与安全执行闭环**

不同于硬编码脚本或固定模板，MorphSheet 将用户的中文自然语言指令实时转化为 Pandas 代码，并在受限沙箱中安全执行。LLM 生成 → AST 扫描 → 沙箱执行 → 失败重试 → 脏数据拦截，形成完整的**生成-验证-执行-纠错**闭环。

**（2）三层脏数据检测机制**

```
预扫描 (LLM 前)  →  捕获源数据中的明显非法值, 使用原始行号+真实值
NaN/NaT 扫描     →  捕获被 errors='coerce' 静默丢弃的值
格式不匹配扫描    →  分析列主导格式, 标记不符合的值 (即使LLM保留了原值)
```

三层互补，确保无论 LLM 代码采用何种策略（强制转换/保留原值/跳过），都不会有脏数据漏网。预扫描还解决了"坏行被后续过滤操作（如删除空行）掩盖"的边界情况。

**（3）技能记忆与进化系统**

系统将一次成功的转换逻辑（源 Schema + Pandas 代码 + 目标格式）打包存入 SQLite，同时通过 ChromaDB 将 Schema 文本向量化。后续上传文件时自动匹配相似技能，匹配成功即可**跳过 LLM 推理，直接执行保存的代码**——实现 0 Token 消耗、毫秒级转换。这赋予了系统"越用越快、越用越准"的进化能力。

**（4）ID 匹配的精确行追踪**

在多步骤转换中（drop_duplicates → dropna → 类型转换），行索引不断变化。系统通过 CUST_ID/员工编号等标识列进行 ID 级匹配，确保：
- Diff 视图中被删行精确定位（不限于末尾）
- 脏数据"跳过"操作精确删除目标行（不误伤相邻行）
- 预扫描用原始行号，后置扫描用结果行号，通过 ID 映射统一

### 4.2 应用创新点

**（1）桌面端零依赖部署**

纯 Python + HTML/CSS/JS 技术栈，无需 Docker、无需 Node.js、无需 GPU。用户安装 Python 3.10+ 后 `pip install -r requirements.txt` 即可运行。Vue.js 通过首次启动自动下载到本地，解决了国内 CDN 不可达问题。

**（2）对话式迭代优化**

系统支持"转换 → 查看 Diff → 不满意 → 继续对话补充指令 → 重新执行"的迭代工作流。对话历史自动累积（`conversationHistory`），每次重试都包含之前的全部指令上下文，让 LLM 了解完整的修正需求。

**（3）老旧系统格式兼容**

专门处理 xls 65535 行限制、CSV GBK 编码 BOM 头、单元格字符截断等实际老旧系统的兼容性问题。这些细节往往是真实企业场景中的"最后一公里"痛点。

### 4.3 难点突破

**（1）LLM 生成代码的安全性**

动态生成的代码可能包含 `os.system("rm -rf /")` 或读取隐私文件。通过 AST 静态扫描 + 受限命名空间 + 超时控制三重防护，确保恶意代码无法执行。

**（2）多格式数据的完整覆盖**

早期版本 LLM 只看到 `head(5)` 的样本数据，仅包含 2/4 种日期格式，导致转换不完整。通过 Schema 格式模式分析（`column_patterns`）+ 样本扩大到 15 行 + Prompt 强化，确保 LLM 了解所有格式变体。

**（3）脏数据检测的误报与漏报平衡**

最初只检测 NaN/NaT，但 LLM 使用 `errors='coerce'` 会静默丢弃坏值。改为禁止 `coerce` 后，坏值被保留原值——但 NaN 检测又失效了。经过多轮迭代，最终建立三层互补检测机制，兼顾了"保留原值"和"检测异常"。

**（4）行号空间的一致性问题**

预扫描用原始行号、后置扫描用结果行号、Diff 用 ID 匹配行号——三个"行号空间"如果不统一，会导致"跳过脏数据"时误删正常行（如 CUST10001 被误删的 bug）。最终通过 `actionable` 字段区分 + ID 匹配执行，彻底解决。

## 五、项目前景与推广建议

### 5.1 应用前景

MorphSheet 可广泛应用于以下场景：

- **企业 IT 部门**：ERP/CRM/HR 系统间的定期数据迁移和格式转换
- **财务部门**：银行对账单、发票数据的清洗和统一格式输出
- **供应链管理**：供应商报价单、库存报表的多格式整合
- **数据分析师**：数据预处理阶段的快速清洗和格式标准化
- **政府/事业单位**：老旧系统（GBK 编码、.xls 格式）的数据升级和归档
