# 07 — 测试与验收

## 一、测试策略

分层测试，自底向上：

| 层级 | 覆盖范围 | 工具 | 目标覆盖率 |
|------|---------|------|-----------|
| 单元测试 | 各模块纯函数 | pytest | 80%+ |
| 集成测试 | API 端点 + Agent 流程 | pytest + httpx | 关键路径全覆盖 |
| 验收测试 | 端到端场景 | 手动 + 两份真实测试数据 | 通过验收标准 |

## 二、测试数据构造

### 测试数据 A：模拟现代 ERP 系统导出（.xlsx）

构造一个包含常见数据问题的 ERP 员工导出表：

```
文件名: erp_employee_export.xlsx
行数: 200 行
```

| 列名 | 类型 | 说明 |
|------|------|------|
| 员工编号 | str | 正常 |
| 姓名 | str | 正常 |
| 性别 | str | 正常 |
| 部门 | str | 正常 |
| 入职日期 | str | 多种格式混合：2023/01/15, 2023-01-15, 2023年1月15日 |
| 联系电话 | str | 含多种格式：010-12345678, 01012345678, (010)12345678 |
| 基本工资 | str | 含"¥"前缀、千分位逗号："¥12,000", "¥8,500" |
| 邮箱 | str | 正常，但有5行缺失（NaN） |
| 备注 | str | 正常，但有2行包含异常长文本（>1000字符） |
| 地址 | str | 完整地址："北京市海淀区中关村大街1号" |

**构造要点**：
- 50% 的"入职日期"使用 `YYYY/MM/DD`，30% 使用 `YYYY-MM-DD`，15% 使用 `YYYY年MM月DD日`，5% 为非法日期
- 30% 的"联系电话"包含连字符，20% 包含括号，10% 包含空格
- 所有"基本工资"包含 `¥` 符号
- 5 行"邮箱"为空白
- 2 行"备注"超过 1000 字符

### 测试数据 B：模拟老旧 CRM 系统导出（.csv, GBK 编码）

构造一个不同结构的客户信息表：

```
文件名: legacy_crm_export.csv
编码: GBK
行数: 500 行
```

| 列名（中文） | 类型 | 说明 |
|-------------|------|------|
| 客户ID | str | 正常 |
| 客户名称 | str | 正常 |
| 行业类别 | str | 正常 |
| 联系人 | str | 20行缺失 |
| 联系电话 | str | 正常，但有重复行干扰 |
| 签约金额 | str | 正常数字 |
| 签约日期 | str | `YYYYMMDD` 无分隔符格式，有3行为非法日期 |
| 客户等级 | str | A/B/C/D，但5行为小写 a/b/c/d |
| 跟进记录 | str | 含大量换行符和特殊字符 |

**构造要点**：
- 文件编码为 GBK（使用 Python `df.to_csv(encoding='gbk')` 生成）
- 10 行完全重复（故意制造的重复数据）
- 3 行"签约日期"为 `99999999` 等非法值
- "联系电话"列有 8 行是相同的（重复数据）
- "跟进记录"包含 `\r\n` 换行，用双引号包裹

### 目标格式 C：统一导入模板（.xls, Excel 97-2003）

模拟一个老旧 ERP 系统的导入要求：

```
格式: .xls (Excel 97-2003)
表头（严格顺序）:
  Emp_ID, Emp_Name, Gender, Department, Join_Date, Phone, Salary, Email, Notes
  （或无客户相关字段的另一个版本）
日期格式: YYYYMMDD (无分隔符)
电话格式: 纯数字，无任何分隔符
薪资格式: 纯数字，无货币符号
```

## 四、集成测试

使用 `pytest-asyncio` + `httpx.AsyncClient` 测试 API 端点：

```python
# tests/conftest.py
import pytest
from app.main import app
from httpx import AsyncClient, ASGITransport

@pytest.fixture
async def client():
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as c:
        yield c
```

关键集成测试场景：

1. **完整上传-转换-导出流程**（使用测试数据 A）
   - 上传 xlsx → 获取 file_id
   - 发送指令："将入职日期统一为 YYYYMMDD 格式，清除电话号码中的分隔符，去掉基本工资的 ¥ 符号"
   - 验证返回 preview 和 diff
   - 确认导出 xls
   - 验证输出文件可读且数据正确

2. **脏数据拦截流程**
   - 上传含非法日期的文件
   - 发送指令："将签约日期转换为 YYYYMMDD"
   - 验证返回 status = "awaiting_human_confirmation"
   - 验证 issues 包含非法日期的行
   - 发送 confirm-action
   - 验证继续执行并成功

3. **技能保存与匹配**
   - 完成一次转换
   - 保存为技能
   - 上传相同 Schema 的文件
   - 验证 match-skills 返回匹配
   - 使用技能 ID 直接转换
   - 验证跳过 LLM 调用（检查 token 消耗为 0，但代码正常执行）

4. **格式降级**
   - 读取 xlsx，转换为 xls
   - 验证输出 xls 格式正确
   - 读取 xlsx，转换为 GBK CSV
   - 验证输出 CSV 编码为 GBK

## 六、验收操作演示脚本

验收时需要演示的完整流程：

### 场景 1：ERP 数据 → 统一模板

1. 启动应用
2. 上传 `erp_employee_export.xlsx`
3. 查看左侧 Schema 面板确认信息正确
4. 设置目标格式为 `.xls`，目标编码默认
5. 在对话区输入：
   > 1. 将入职日期统一为 YYYYMMDD 格式
   > 2. 清除联系电话中的所有分隔符（括号、连字符、空格）
   > 3. 去掉基本工资的 ¥ 符号和千分位逗号，转为纯数字
   > 4. 表头映射：员工编号→Emp_ID, 姓名→Emp_Name, 性别→Gender, 部门→Department, 入职日期→Join_Date, 联系电话→Phone, 基本工资→Salary, 邮箱→Email
6. 观察右侧状态面板显示 Agent 步骤
7. 查看 Diff 视图确认转换结果
8. 确认导出
9. 保存为技能："ERP员工表→统一模板"
10. 验证输出 xls 文件可正常打开且数据正确

### 场景 2：CRM 数据 → 统一模板（含脏数据）

1. 上传 `legacy_crm_export.csv`（GBK 编码）
2. 设置目标格式为 `.csv`，编码 `utf-8`
3. 在对话区输入：
   > 1. 去除所有重复行
   > 2. 将客户等级统一为大写
   > 3. 签约日期从 YYYYMMDD 转为 YYYY-MM-DD
   > 4. 删除联系人为空的行
4. 观察 Agent 遇到 `99999999` 非法日期时暂停
5. 在弹窗中选择"跳过此行"
6. 查看 Diff 确认结果
7. 确认导出
8. 验证输出的 CSV 已去除重复行且日期格式正确
9. 确认之前保存的技能在匹配列表中显示
