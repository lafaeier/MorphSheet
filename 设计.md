# 跨系统表格数据转换智能体 - 系统设计文档
## 一、 前端页面设计（UI/UX）
**设计理念**：采用当下主流的 **“极简工具+AI对话”** 范式（参考 Notion AI / Cursor / Dify），以深色/浅色双模为基调，强调数据流的透明感与操作的区域划分。
**布局结构（三栏式核心布局）**：
1.  **左侧边栏**：历史与会话区
    *   **转换历史**：按时间倒序排列过往任务。
    *   **技能库**：可视化的工作流/模板列表。展示已沉淀的技能卡片，支持一键拖拽应用到主工作区，无需再次自然语言描述。
2.  **中间主工作区**：数据与交互核心
    *   **顶部操作栏**：源文件拖拽上传区（高亮支持的格式 `.xlsx/.xls/.csv`）、目标格式下拉选择器、执行按钮。
    *   **核心对话区**：浮动式 AI 对话框，用户在此输入自然语言清洗指令，AI 回复执行状态与异常确认。
    *   **Diff 视图（加分项核心）**：左右分屏对比视图。左屏“源数据预览”，右屏“转换后预览”。数据修改处采用类似 Git 的红绿高亮，新增列标绿，删除行标红，修改单元格标黄，确保透明度。
3.  **右侧状态栏**：Agent 思考与日志区
    *   实时显示 Agent 的思维链（如：正在分析Schema -> 正在映射字段 -> 正在生成Pandas代码 -> 沙箱执行中）。
    *   **异常阻断弹窗**：当遇到脏数据，此区域变红并弹出高亮异常行及修改建议，需用户点击“确认/修改”后 Agent 才能继续。

## 二、 系统执行流设计
这是系统的中枢神经，核心在于 **“意图转代码，执行必沙箱，沉淀为逻辑”**。
### 核心执行流（从输入到输出）
**Step 1: 接收与Schema提取（输入端）**
*   **输入**：用户上传源文件 + 指定目标格式规范 +（可选）自然语言指令/选择技能模板。
*   **动作**：系统使用 Pandas 读取源文件的前 N 行及表头，提取 Schema（列名、数据类型样本），并将其序列化为 JSON 交给 LLM。
**Step 2: 意图理解与逻辑规划（大脑端）**
*   **动作**：LLM 结合源 Schema、目标要求和用户自然语言，进行多步推理：
    1.  **语义映射**：计算源列与目标列的对应关系（如“姓名”->Emp_Name）。
    2.  **操作翻译**：将自然语言（“剔除金额<0”）翻译为具体的 Pandas 操作步骤。
    3.  **代码生成**：将映射和操作合并，动态生成一段完整的 Python/Pandas 处理代码。
**Step 3: 沙箱执行与异常捕获（别出心裁的设计：安全兜底与人机协同）**
*   **动作**：将生成的代码放入受限的沙箱环境（如 Docker 或 RestrictedPython）中执行。
*   **异常分支（脏数据拦截）**：如果代码在执行某行时因数据格式错乱报错（如尝试把非数字转为整数），Agent 捕获该异常，**挂起执行**，向前端发送状态 `awaiting_human_confirmation`，高亮报错行与建议，等待人工决策。
*   **自纠错分支（代码Bug修复）**：如果是 LLM 生成的代码本身语法错误，Agent 会将报错信息喂给 LLM，让其重新生成代码（最多重试3次）。
**Step 4: 预览与对齐（输出端前置）**
*   **动作**：代码执行成功后，生成目标 DataFrame。此时**不立即写盘**，而是将内存中的数据渲染到前端的“Diff 视图”供用户确认。
**Step 5: 格式降级与导出（输出端）**
*   **输入**：用户在 Diff 视图点击“确认导出”。
*   **动作**：系统根据目标格式要求，处理底层兼容性问题（如：处理 .xls 的65536行截断、注入 CSV 的 GBK 编码 BOM 头、强制转换日期格式字符串等），最终落盘输出文件。
**Step 6: 工作流沉淀（别出心裁的设计：记忆进化）**
*   **动作**：用户确认导出成功后，Agent 发起询问“是否保存为技能？”。若保存，系统将 **【源Schema特征 + 目标Schema要求 + LLM本次生成的Pandas代码】** 打包存入本地向量数据库/JSON库。
*   **价值**：下次遇到同源文件，系统匹配到特征后，直接调取历史代码执行，**跨越 LLM 推理环节，0 Token消耗，毫秒级转换**。

## 三、 技术栈推荐
基于 Windows 桌面端、Python 数据处理生态及 AI Agent 架构，推荐以下现代化技术栈：
### 1. 客户端桌面框架 (GUI)
*   **推荐：Tauri + Vue 3 / React**
    *   *理由*：相比笨重的 Electron，Tauri 基于 Rust，打包体积极小（几MB），性能极高，极其适合需要频繁处理本地文件的桌面端应用。前端使用 Vue/React 可以完美实现现代极简 UI。
*   *备选*：PySide6 / PyQt6 (纯 Python 方案，开发快但 UI 现代化成本高)。
### 2. 后端核心与数据处理引擎
*   **语言**：Python 3.10+
*   **Web框架**：FastAPI (负责与 Tauri 前端进行本地 HTTP/WebSocket 通信，推送 Agent 状态)
*   **数据处理**：Pandas, OpenPyXL (处理 xlsx), XlsxWriter, xlrd/xlwt (处理老旧 xls)
*   **编码检测**：chardet (自动识别源 CSV 的乱码编码)
### 3. AI Agent 与 LLM 架构
*   **Agent 框架**：LangChain 或 LlamaIndex (用于构建思维链、管理 Prompt 模板、实现重试逻辑)
*   **LLM 模型选择**：
    *   云端：GPT-4o / Claude-3.5-Sonnet （代码生成能力极强，适合复杂映射）
    *   本地：Qwen2.5-Coder / DeepSeek-Coder （保障企业数据不出网）
*   **输出约束**：严格使用 LLM 的 Structured Output / JSON Mode 功能，确保生成的代码和状态标识易于解析。
### 4. 沙箱安全执行环境
*   **推荐：Docker API (本地轻量级容器)**
    *   *理由*：动态生成的代码具有不可控性。每次执行代码时，启动一个微型 Docker 容器，挂载源文件，执行完毕后销毁。彻底防止 LLM 生成恶意代码（如 `os.system("rm -rf /")` 或读取用户隐私）。
*   *备选*：RestrictedPython (轻量级 Python 沙箱，安全性不如 Docker，但无需安装 Docker 环境)。
### 5. 本地记忆与工作流存储
*   **关系数据**：SQLite (存储工作流元数据、历史记录)
*   **向量检索**：ChromaDB / FAISS (存储源 Schema 的 Embedding，用于后续上传文件时快速匹配相似的历史技能模板)
